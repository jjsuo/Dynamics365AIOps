# 第三层：飞书协作层

核心抽象——**话题(thread) = 并行推进的最小治理单元**——按几人小团队大幅简化：不自建 runtime、不做多引擎调度，只用飞书话题 + 一个轻量通知脚本。

## 核心抽象
一个**模块/需求 = 一个飞书话题**。这个话题就是一条独立交付链路，自带四个边界：

| 边界 | 在话题里如何体现 |
|---|---|
| 上下文 | 该模块的需求、FDD、讨论都在这个话题里 |
| 参与者 | 谁在话题里 = 谁对这个模块负责 |
| 决策 | 门禁卡片的 `yes`/`e` 回复就是决策记录 |
| 状态 | 卡片上的阶段(P03/S07/...)就是当前状态 |

## 工作方式
1. 一个**工作台群**，每个模块在群里开一个话题。
2. DAF 流程跑到 pending_review（P03/P05/S06/S07/T01）时，`notify_feishu.py` 往对应话题发一张**门禁卡片**（阶段 + 产物摘要 + 放行指引）。
3. 顾问/架构师在手机或桌面，在话题里回复 `yes` 放行 / `e + 意见` 退回——**异步、跨设备**。
4. 放行/退回的结论 + 讨论 **回写**到需求池 Hub 对应产物的 commit message 或 `decisions/<id>.md`，成为项目资产。

## 三个增量阶段（按需做）
- **MVP（已提供）**：发门禁卡片到话题 + 人工回复放行。`notify_feishu.py` 即可。
- **增量 1**：读话题回复（`im/v1/messages` list + thread_id）自动判定 yes/e，回写状态。
- **增量 2**：卡片按钮放行（需配置事件回调 URL + 卡片 action；注意飞书按钮回调要额外开通，否则点击报 200340）。小团队非必需。

## 与 DAF 的衔接
`daf_starter.md` / `s_starter.md` 每到门禁阶段，调用 `collab/notify_feishu.py`；放行后再继续下一阶段。把"yes 放行"从本地终端搬到飞书话题，让不在电脑前的人也能决策。
